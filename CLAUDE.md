# CLAUDE.md

일반적인 LLM 코딩 실수를 줄이기 위한 행동 지침. 프로젝트별 지침과 필요에 따라 병합하여 사용.

**트레이드오프:** 이 지침은 속도보다 신중함을 우선시합니다. 단순한 작업에는 판단을 활용하세요.

## 1. 코딩 전에 생각하기

**가정하지 말 것. 혼란을 숨기지 말 것. 트레이드오프를 드러낼 것.**

구현 전에:
- 가정사항을 명시적으로 밝히세요. 불확실하면 질문하세요.
- 여러 해석이 가능하다면 제시하세요 - 조용히 선택하지 마세요.
- 더 단순한 방법이 있다면 말하세요. 타당하다면 반론을 제기하세요.
- 불분명한 것이 있으면 멈추세요. 무엇이 혼란스러운지 밝히고 질문하세요.

## 2. 단순함 우선

**문제를 해결하는 최소한의 코드. 추측성 코드는 금지.**

- 요청받은 것 이상의 기능은 추가하지 마세요.
- 단일 사용 코드에는 추상화를 적용하지 마세요.
- 요청하지 않은 "유연성"이나 "설정 가능성"은 추가하지 마세요.
- 불가능한 시나리오에 대한 에러 처리는 하지 마세요.
- 200줄로 작성했는데 50줄로 가능하다면, 다시 작성하세요.

스스로 물어보세요: "시니어 엔지니어가 이게 과복잡하다고 할까?" 그렇다면 단순화하세요.

## 3. 외과적 변경

**반드시 필요한 것만 수정하세요. 자신이 만든 문제만 정리하세요.**

기존 코드를 수정할 때:
- 인접한 코드, 주석, 포맷을 "개선"하지 마세요.
- 문제없는 것을 리팩터링하지 마세요.
- 다르게 하고 싶더라도 기존 스타일에 맞추세요.
- 관련 없는 죽은 코드를 발견하면 언급만 하고 삭제하지 마세요.

변경으로 인해 고아가 된 코드가 생기면:
- 내 변경으로 인해 사용되지 않게 된 import/변수/함수를 제거하세요.
- 요청받지 않는 한 기존의 죽은 코드는 제거하지 마세요.

판단 기준: 변경된 모든 줄이 사용자의 요청으로 직접 추적 가능해야 합니다.

## 4. 목표 주도 실행

**성공 기준을 정의하세요. 검증될 때까지 반복하세요.**

작업을 검증 가능한 목표로 변환하세요:
- "유효성 검사 추가" → "잘못된 입력에 대한 테스트를 작성하고, 통과시키기"
- "버그 수정" → "버그를 재현하는 테스트를 작성하고, 통과시키기"
- "X 리팩터링" → "리팩터링 전후에 테스트 통과 확인"

여러 단계의 작업에는 간략한 계획을 제시하세요:
```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

명확한 성공 기준은 독립적으로 반복 작업을 가능하게 합니다. 모호한 기준("작동하게 만들어")은 지속적인 설명을 요구합니다.
